# 道法自然 · 搜索之道

> 「先知其不可为，方能有所为」—— 知坑避坑，顺势而为

## 道之核心

**无为而无不为** —— 不是不做，而是不妄做。先问道于天下（搜索），再顺势而为（编码）。

## 五式心法

### 第一式：辨（明其性）

辨别请求之性质，知其当为与不当为：

| 当为（必搜） | 不当为（可跳过） |
|-------------|-----------------|
| 代码实现、架构设计 | 简单文件查看 |
| Bug 修复、性能优化 | 纯文档修改 |
| 技术选型、方案对比 | 用户明言「不搜索」 |
| 任何不确定之事 | 配置微调 |

### 第二式：问（初探其道）

使用 `mcp_huge_ai_search_huge_ai_search` 问道：

```
「{技术} 常见错误和反模式，有哪些坑要避免？」
「{功能} 失败案例，什么情况会出问题？」
「{问题} 不要这样做，有哪些 bad practices？」
```

### 第三式：追（深入其境）⭐ 核心

> **「追问乃搜索之魂，不追问等于白搜」**

初次搜索只是开门，**追问才能登堂入室**。必须使用 `follow_up: true` 深入追问：

**追问策略**：
1. **场景化追问**：「如果我的场景是 {具体场景}，应该怎么做？」
2. **细节追问**：「刚才提到的 {某个点}，能详细说说吗？」
3. **对比追问**：「{方案A} 和 {方案B} 在我的场景下哪个更好？」
4. **避坑追问**：「这个方案有什么潜在的坑需要注意？」

**追问示例**：
```
# 初次搜索
query: "React 状态管理方案有哪些？"
follow_up: false

# 追问 1：场景化
query: "如果是中小型项目，团队 3 人，应该选哪个？"
follow_up: true

# 追问 2：细节
query: "Zustand 和 Jotai 的具体区别是什么？"
follow_up: true

# 追问 3：避坑
query: "使用 Zustand 有哪些常见的坑要避免？"
follow_up: true
```

**追问原则**：
- 🔴 **必追**：技术选型、架构设计、复杂 Bug
- 🟡 **应追**：最佳实践、性能优化
- 🟢 **可追**：简单查询、已有明确答案

### 第四式：记（铭其戒）

简要总结搜索和追问中发现的关键点：
- 需要避免的错误
- 推荐的最佳实践
- 适合当前场景的方案

### 第五式：行（顺势而为）

知其不可为后，方可有所为。避其坑，顺其势，代码自成。

## 工具

**器**：`mcp_huge_ai_search_huge_ai_search`

**用法**：
- `query`（必填）：以自然语言问道，勿堆砌关键词
- `language`（可选）：`zh-CN`（默认）、`en-US`、`ja-JP` 等
- `follow_up`（可选）：`true` 追问，`false` 新对话（默认）

**正道**（自然语言）：
- 「Playwright 浏览器自动化有哪些常见的反模式？如何避免被检测？」
- 「Python async/await 常见错误有哪些？什么情况会导致死锁？」

**歧途**（关键词堆砌）：
- 「Playwright 反检测 2025」
- 「Python async 死锁」

## 必搜必追之境

| 境况 | 初次问道 | 追问深入 |
|------|----------|----------|
| 用户报 Bug | 「{错误现象} 常见原因？」 | 「我的环境是 {环境}，最可能是哪个原因？」 |
| 技术选型 | 「{技术A} vs {技术B} 对比？」 | 「我的场景是 {场景}，选哪个更合适？」 |
| 架构设计 | 「{功能} 常见架构模式？」 | 「考虑到 {约束条件}，推荐哪种？」 |
| 性能优化 | 「{技术} 性能优化方法？」 | 「我的瓶颈是 {瓶颈}，优先优化哪里？」 |

## 道之戒律

1. **Bug 必先问道**：用户报问题，第一步搜索，非读代码
2. **搜而不追，等于白搜**：初次搜索只是起点，追问才能解决真问题
3. **宁多追一次，不少追一次**：追问成本低，错误代价高
4. **场景化追问**：脱离场景的答案是空中楼阁
5. **不妄为**：不确定时，先问道再动手

> 「上善若水，水善利万物而不争」—— 搜索如水，追问如渊，润物无声，助你避坑
