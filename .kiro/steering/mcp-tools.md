# 道法自然 · 搜索之道

> 「先知其不可为，方能有所为」—— 知坑避坑，顺势而为

## 道之核心

**无为而无不为** —— 不是不做，而是不妄做。先问道于天下（搜索），再顺势而为（编码）。

## 三式心法

### 第一式：辨（明其性）

辨别请求之性质，知其当为与不当为：

| 当为（必搜） | 不当为（可跳过） |
|-------------|-----------------|
| 代码实现、架构设计 | 简单文件查看 |
| Bug 修复、性能优化 | 纯文档修改 |
| 技术选型、方案对比 | 用户明言「不搜索」 |
| 任何不确定之事 | 配置微调 |

### 第二式：避（知其不可为）

**先知其不可为，方能有所为。**

使用 `mcp_google_ai_search_google_ai_search` 问道：

```
「{技术} 常见错误和反模式，有哪些坑要避免？」
「{功能} 失败案例，什么情况会出问题？」
「{问题} 不要这样做，有哪些 bad practices？」
```

### 第三式：行（顺势而为）

知其不可为后，方可有所为。避其坑，顺其势，代码自成。

## 工具

**器**：`mcp_google_ai_search_google_ai_search`

**用法**：
- `query`（必填）：以自然语言问道，勿堆砌关键词
- `language`（可选）：`zh-CN`（默认）、`en-US`、`ja-JP` 等

**正道**（自然语言）：
- 「Playwright 浏览器自动化有哪些常见的反模式？如何避免被检测？」
- 「Python async/await 常见错误有哪些？什么情况会导致死锁？」

**歧途**（关键词堆砌）：
- 「Playwright 反检测 2025」
- 「Python async 死锁」

## 必搜之境

| 境况 | 问道示例 |
|------|----------|
| 用户报 Bug | 「{错误现象} 常见原因和解决方案？」 |
| 浏览器自动化 | 「Patchright/Playwright 反检测最佳实践？」 |
| 网页抓取 | 「Google 搜索结果抓取，如何绕过反爬？」 |
| 异步编程 | 「Python asyncio 常见陷阱和避坑指南？」 |
| MCP 开发 | 「MCP SDK tool 实现有哪些常见错误？」 |

## 道之戒律

1. **Bug 必先问道**：用户报问题，第一步搜索，非读代码
2. **宁多搜一次，不少搜一次**：搜索成本低，错误代价高
3. **不妄为**：不确定时，先问道再动手

> 「上善若水，水善利万物而不争」—— 搜索如水，润物无声，助你避坑
